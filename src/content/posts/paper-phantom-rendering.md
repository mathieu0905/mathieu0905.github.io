# Phantom Rendering：揪出 UI 中看不见的性能杀手

> 📄 发表于 **FSE 2026**
>
> 作者：Zhihao Lin, Mingyi Zhou, Han Hu, Bo Sun, Gang Fan, Li Li

---

## 一句话概括

我们发现移动应用中普遍存在"幽灵渲染"——UI 框架在计算用户永远看不到的像素，白白消耗 GPU 和电量。

---

## 什么是 Phantom Rendering？

想象你在用一个 App，屏幕上显示着一个列表。在列表的下方（屏幕外），可能还有几十个 item 已经被渲染好了，占用着 GPU 资源——但你根本看不到它们，因为它们不在可视区域内。

这就是 **Phantom Rendering（幽灵渲染）**：
- UI 组件被创建和渲染了，但从未出现在用户可见的区域
- 渲染计算确实发生了（消耗了 CPU/GPU），但产生的像素永远不会被显示
- 在复杂的 UI 层级嵌套中，这种情况比你想象的更普遍

## 为什么之前没人关注这个？

传统的性能分析工具关注的是"哪里慢"——卡顿、掉帧、长耗时操作。但 Phantom Rendering 不一定会直接导致明显的卡顿，它更像是一个**隐性的性能税**：

- 每一帧多渲染几个不可见的组件，可能只慢了几毫秒
- 但累积起来，会影响帧率稳定性、增加功耗、缩短电池寿命
- 在低端设备上，影响尤其明显

## 我们怎么检测的？

### 核心思路
对比"UI 框架认为需要渲染的区域"和"用户实际能看到的区域"，二者的差集就是 Phantom Rendering。

### 技术方案
1. **渲染树分析**：解析应用的 UI 渲染树，获取每个组件的布局位置和尺寸
2. **可见性计算**：根据屏幕尺寸、滚动位置、遮挡关系，计算每个组件的实际可见性
3. **渲染追踪**：监控 GPU 的渲染调用，记录哪些组件确实触发了绘制操作
4. **交叉分析**：不可见但被渲染的组件 = Phantom Rendering

### 配套工具
我们构建了 [HapRay](https://github.com/ArkAnalyzer-HapRay) 工具来自动化这个检测过程，开发者可以直接在自己的 App 上跑分析。

## 关键发现

- 在我们测试的应用中，Phantom Rendering 普遍存在
- 部分场景下，不可见的渲染计算占总渲染负载的相当比例
- 修复这些问题后，帧率稳定性和功耗都有可度量的改善

## 对开发者的建议

- **懒加载**：只在组件即将进入可视区域时才创建和渲染
- **回收机制**：离开可视区域的组件及时回收渲染资源
- **审计工具**：定期用性能分析工具检查是否存在不必要的渲染
